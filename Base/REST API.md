Глалгол метода запроса(GET, POST, DELETE, PUT)
Существительное - URI запроса
Дополнительная информация - хедеры запроса.
Содержимое - тело запроса и ответ JSON
Результат - код ответа.

- Основные ошибки
	1. Внутри GET запроса на стороне сервера нельзя менять, удалять данные. # GET запросы кешируются. Вызывая andpoint или ручку бизнес логика не будет выполнена так как запрос будет получен из кеша.
	2. PUT & PATCH & POST.  POST - используется для добавления новой сущности(загрузить фаил, добавить запись в БД, создать заявление). PUT - используется чтобы изменить уже существующую сущность на сервере.(Изменить уже имеющуюся заптсь в БД, изменить фаил, изменить запись в кеше). PATCH -используется дляизменения малой части сущности.(Атрибут, строка, столбец в нереляционной базе данных).# Зачастую в PUT передается много данных. PATCH менее подвержен ошибкам со стороны клиента.
	3. Нейминг эндпоиндов. Часто разработчики используют единственное число (/user, /product). Но более правильно использовать множественное число(/users, /products). Если перейти по этой ручке то можно полчить полный список продуктов или пользователей. Для получения конкретного нужно после ключевого слова написать id (.../users/id) .
	4. Иерархия. Нужно избегать большой иерархии данных и не привязывать одни данные к другим.
	5. Версионирование.Нужно добавлять версии энпоинтов и при этом оставлять предыдущие версии в рабочем состоянии.
	6. Передача чувствительных данных в query параметрах - плохая идея. Нельзя передавать API-ключи, пароли, почты, номера телеонов. Их нужно передавать в теле запросов.# Очень часто логируются URl по которым идет обращение (ElasticSerch, Grafana, Loki можно посмотреть) Утечки данных часто происходят при логировании чувствительных данных. Например Twiter.
	7. Не передавать масивы на верхнем уровне ответа. Нельзя отвечать JSON который является массивом(списком). Из -за этого нельзя расширять API. Использовать формат ключ значение.
	8. Работа с датой и временем. Использовать количество милисекунд с 1970 года или использовать в формате ISO 8601 в котором есть временная зона и удобство работы как разработчику.
