Это структура Backend приложения отвечающая, отвечающая за взаимодействие внутренних частей программы, обеспечивая функциональность приложения.

| Архитектура                       | Плюсы                                                                                              | Минусы                                                                                    | Когда выбирать                                               |
| --------------------------------- | -------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| **[[Монолит]]**                   | Просто начать, одна БД и деплой, легко отлаживать                                                  | Плохо масштабируется кусочно, релизы блокируют друг друга, со временем становится «комом» | Стартапы, маленькие команды, быстрый MVP                     |
| **[[Модульный монолит]]**         | Чёткая структура, проще эволюционировать в микросервисы, всё ещё один деплой                       | Всё равно общий релиз, масштабирование не по частям                                       | Малые/средние проекты, которые могут расти                   |
| **[[Микросервисы]]**              | Независимые релизы, точечное масштабирование, изоляция отказов, команды могут работать параллельно | Сложность DevOps, наблюдаемости, согласованности данных, сетевые задержки                 | Крупные системы, много команд, разные темпы релизов          |
| **[[SOA]]**                       | Повторное использование сервисов, выделение бизнес-функций                                         | Тяжёлый ESB, меньше гибкости, устаревающий подход                                         | Энтерпрайз-решения «старой школы», интеграции                |
| **[[Serverless FaaS]]**           | Авто-масштабирование, плата за использование, нет серверов для поддержки                           | Холодные старты, ограничения по времени/ресурсам, сложно хранить состояние                | Прототипы, событийные системы, нерегулярные пиковые нагрузки |
| **[[Event-driven]]**              | Асинхронность, слабая связность, высокая масштабируемость                                          | Сложный дебаг, eventual consistency (не сразу согласованные данные)                       | Микросервисы с большим обменом событиями, интеграции         |
| **[[CQRS]] + [[Event Sourcing]]** | Масштабирование чтения/записи, история изменений, аудит                                            | Сложнее проектировать и поддерживать, не всегда нужно                                     | Финтех, логистика, системы с критичной историей данных       |
| **[[Layered (3-tier)]]**          | Понятная, классическая структура (UI → логика → БД)                                                | Жёсткая, плохо адаптируется под рост нагрузки                                             | Учебные проекты, небольшие корпоративные системы             |

Дерево Выбора Архитектуры:

- **Размер команды и скорость развития**
    
    - ≤ 10 разработчиков, нужен быстрый старт → **Монолит / Модульный монолит**
        
    - > 10 разработчиков, разные темпы релизов → переходи к шагу 2
        
- **Нагрузка и масштабирование**
    
    - Нагрузка небольшая или равномерная → можно остаться в **модульном монолите**
        
    - Есть «горячие зоны» (поиск, платежи, аналитика) → подумай о **микросервисах**
        
- **Критичность отказов**
    
    - Отказ одного модуля недопустим (например, платежи не должны ронять каталог) → **Микросервисы / Event-driven**
        
    - Отказ всего приложения терпим → оставайся в **монолите**
        
- **Технологическая разнородность**
    
    - Вся команда работает на одном стеке → монолит/модульный монолит ок
        
    - Нужны разные технологии (Go для API, Python для ML, Java для транзакций) → **Микросервисы**
        
- **Характер задач**
    
    - Много асинхронных событий, фоновые задачи → добавь **Event-driven** слой
        
    - Нерегулярные, спорадические вызовы (например, только при апдейте данных) → **Serverless**
        
    - Требуется аудит и точная история изменений → **CQRS + Event Sourcing**
